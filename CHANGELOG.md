Uzrok — isti obrazac kao Dan 44 REVOKE zamka: trigger funkcija (`enforce_event_interaction_rate_limit`) bila je obična SECURITY INVOKER plpgsql funkcija, pa se njena vlastita `select count(*) from event_interactions` izvršavala s privilegijama pozivatelja (anon); anon nema NIKAKVU SELECT politiku na `event_interactions` (samo INSERT, ADR-014/0009), pa je RLS triggeru uvijek vraćao 0 redaka bez obzira koliko ih stvarno postoji — limit se nikad nije mogao dosegnuti. `supabase/migrations/0019_fix_rate_limit_trigger_rls_blind.sql` — funkcija postaje `SECURITY DEFINER` (isti obrazac kao popularity/trending/view_count funkcije), broji retke s privilegijama vlasnika bez izlaganja redaka pozivatelju. Uživo re-testirano: točno 20 anon insertova uspijeva, 21. odbijen s očekivanom porukom, stvaran broj redaka u bazi = 20, normalno praćenje jednog pregleda (`ViewTracker`) i dalje radi bez greške. Vidi DECISIONS.md ADR-018.
- **Faza 8, Dan 44 (ispravak) — column-level REVOKE iz 0014 nije radio:** live provjera (service-role + anon test skripta protiv stvarne baze nakon što je korisnik primijenio 0014) otkrila je da `submitter_email`/`submitter_phone` ostaju čitljivi anonu na objavljenom događaju unatoč `revoke select (submitter_email, submitter_phone) on events from anon;`. Uzrok: poznata Postgres zamka — anon je SELECT na `events` već imao na razini cijele tablice (Supabase automatski dodjeljuje `grant all` pri kreiranju tablice), a column-level REVOKE ne oduzima pristup dodijeljen kroz taj širi table-level grant. `supabase/migrations/0015_fix_events_column_privileges.sql` — `revoke select on events from anon` (cijela tablica) + `grant select (...)` s eksplicitnim popisom svih kolona osim `submitter_email`/`submitter_phone`. Vidi DECISIONS.md ADR-016 dopunu. Ubuduće svaki novi stupac na `events` mora biti ručno dodan na taj grant popis. Migracija čeka ručno lijepljenje u Supabase Dashboard SQL Editoru.
- **Prvi produkcijski build na Vercelu pukao** (`TypeError: Invalid URL`, `Failed to collect page data for /_not-found`) — `NEXT_PUBLIC_SITE_URL` u Vercel Environment Variables bio je postavljen kao gola domena (`kam-ve-iti.vercel.app`) bez `https://` prefiksa; `new URL(...)` u `layout.tsx` (metadataBase) to odbija. Popravljeno postavljanjem pune vrijednosti (`https://kam-ve-iti.vercel.app`) i redeployem. `.env.example` dobio eksplicitno upozorenje o obaveznom protokolu da se ne ponovi.
- GitHub secret scanning blokirao push jer je stvarni Supabase secret key greškom završio u `.env.example` umjesto `.env.local` — očišćeno, ključ nikad nije stigao na remote.
- `.env.local` sadržavao dupliciranu/spojenu verziju varijabli (stari prazan predložak zalijepljen ispod novih vrijednosti) — kod dupliciranih `.env` varijabli zadnja definicija pobjeđuje, pa je prazna verzija odozdo prepisivala stvarni URL. Ubuduće: `.env.local` se puni SAMO stvarnim vrijednostima, bez ostataka predloška.
- **Bug otkriven pri izradi loading skeletona (Faza 8, Dan 3):** `<main>` element unutar `loading.tsx` fallbacka sudara se sa stvarnim `<main>` u stranici koja se streamа — u ovoj verziji Next.js/Turbopack dev servera (16.2.10) to trajno blokira zamjenu fallbacka stvarnim sadržajem (stranica ostaje zaglavljena na skeletonu, potvrđeno uživo kroz više neovisnih testova: čist restart servera, nova kartica, prazan router cache). Rješenje: fallback koristi `<div>` s istim izgledom — ujedno i ispravnije pristupačnosti, jer se izbjegava privremeno dupliciranje `<main>` landmarka.
